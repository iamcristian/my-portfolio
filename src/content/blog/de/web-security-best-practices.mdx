## Warum Sicherheit wichtig ist

Als Entwickler mit einem **Aufbaustudium in Informationssicherheit** habe ich aus erster Hand gesehen, wie selbst kleine Schwachstellen zu katastrophalen Sicherheitsverletzungen führen können. Dieser Beitrag behandelt die wesentlichen Sicherheitspraktiken, die jeder Entwickler implementieren sollte.

### Die OWASP Top 10 (Ausgabe 2025)

Die OWASP Top 10 sind der Goldstandard für Sicherheitsrisiken in Webanwendungen:

1. **Fehlerhafte Zugriffskontrolle** — Die kritischste Schwachstelle
2. **Kryptographische Fehler** — Fehlerhafte Verschlüsselungsimplementierung
3. **Injection** — SQL, NoSQL, LDAP, OS-Befehlsinjektion
4. **Unsicheres Design** — Sicherheitsmängel auf Architekturebene
5. **Sicherheitsfehlkonfiguration** — Standardanmeldedaten, offener Cloud-Speicher
6. **Verwundbare Komponenten** — Veraltete Abhängigkeiten
7. **Authentifizierungsfehler** — Schwache Passwortrichtlinien, Session-Hijacking
8. **Datenintegritätsfehler** — Manipulation der CI/CD-Pipeline
9. **Protokollierungsfehler** — Unzureichende Überwachung
10. **Server-Side Request Forgery** — SSRF-Angriffe

### Eingabevalidierung — Deine erste Verteidigungslinie

**Vertraue niemals Benutzereingaben.** Validiere immer auf Client- und Serverseite:

```typescript
import { z } from "zod";

// Definiere strikte Schemas für alle Eingaben
const CreateUserSchema = z.object({
  email: z.string().email().max(255),
  password: z
    .string()
    .min(12, "Passwort muss mindestens 12 Zeichen haben")
    .regex(/[A-Z]/, "Muss einen Großbuchstaben enthalten")
    .regex(/[a-z]/, "Muss einen Kleinbuchstaben enthalten")
    .regex(/[0-9]/, "Muss eine Zahl enthalten")
    .regex(/[^A-Za-z0-9]/, "Muss ein Sonderzeichen enthalten"),
  name: z
    .string()
    .min(2)
    .max(100)
    .regex(/^[a-zA-ZÀ-ÿ\s'-]+$/),
});

// In deinem Controller/Handler
async function createUser(req: Request) {
  const result = CreateUserSchema.safeParse(await req.json());

  if (!result.success) {
    return new Response(
      JSON.stringify({
        error: "Validierung fehlgeschlagen",
        details: result.error.flatten(),
      }),
      { status: 400 },
    );
  }

  // Sicher result.data zu verwenden
  const { email, password, name } = result.data;
  // ... Benutzer mit gehashtem Passwort erstellen
}
```

### Best Practices für Authentifizierung

#### Verwende bcrypt oder Argon2 für Passwort-Hashing

```typescript
import bcrypt from "bcrypt";

const SALT_ROUNDS = 12;

async function hashPassword(password: string): Promise<string> {
  return bcrypt.hash(password, SALT_ROUNDS);
}

async function verifyPassword(
  password: string,
  hash: string,
): Promise<boolean> {
  return bcrypt.compare(password, hash);
}
```

> **Niemals** Passwörter im Klartext speichern. Niemals MD5 oder SHA für Passwörter verwenden. Verwende immer einen langsamen Hash-Algorithmus, der für Passwörter entwickelt wurde.

#### JWT Best Practices

```typescript
import jwt from "jsonwebtoken";

// ✅ Gut: Kurzlebige Access-Tokens + Refresh-Token-Rotation
const accessToken = jwt.sign(
  { userId: user.id, role: user.role },
  process.env.JWT_SECRET!,
  {
    expiresIn: "15m", // Kurzlebig
    issuer: "cristianarando.app",
    audience: "api",
  },
);

// ✅ Gut: HTTP-only, Secure, SameSite Cookies
response.cookie("refreshToken", refreshToken, {
  httpOnly: true, // Nicht über JavaScript zugänglich
  secure: true, // Nur über HTTPS gesendet
  sameSite: "strict", // CSRF-Schutz
  maxAge: 7 * 24 * 60 * 60 * 1000, // 7 Tage
  path: "/api/auth", // Nur an Auth-Endpoints gesendet
});
```

### API-Sicherheitscheckliste

Beim Erstellen von APIs immer implementieren:

#### Rate Limiting

```typescript
import rateLimit from "express-rate-limit";

// Allgemeines API-Rate-Limit
const apiLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 Minuten
  max: 100, // 100 Anfragen pro Fenster
  standardHeaders: true,
  legacyHeaders: false,
  message: { error: "Zu viele Anfragen, bitte versuche es später erneut." },
});

// Striktes Limit für Auth-Endpoints
const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 5, // Nur 5 Versuche pro 15 Minuten
  skipSuccessfulRequests: true,
});
```

#### Sicherheitsheader

```typescript
import helmet from "helmet";

app.use(
  helmet({
    contentSecurityPolicy: {
      directives: {
        defaultSrc: ["'self'"],
        scriptSrc: ["'self'", "'unsafe-inline'"],
        styleSrc: ["'self'", "'unsafe-inline'", "fonts.googleapis.com"],
        fontSrc: ["'self'", "fonts.gstatic.com"],
        imgSrc: ["'self'", "data:", "https:"],
        connectSrc: ["'self'"],
        frameSrc: ["'none'"],
        objectSrc: ["'none'"],
      },
    },
  }),
);
```

### SQL-Injection-Prävention

**Verwende immer parametrisierte Abfragen:**

```typescript
// ❌ GEFÄHRLICH — SQL-Injection-Schwachstelle
const query = `SELECT * FROM users WHERE email = '${email}'`;

// ✅ SICHER — Parametrisierte Abfrage
const result = await db.query("SELECT * FROM users WHERE email = $1", [email]);

// ✅ SICHER — Verwendung eines ORM (Prisma)
const user = await prisma.user.findUnique({
  where: { email },
});
```

### Sicherheitscheckliste für jedes Projekt

Verwende diese Checkliste vor dem Deployment jeder Anwendung:

- [ ] Alle Eingaben validiert und bereinigt
- [ ] Passwörter mit bcrypt/Argon2 gehasht (Salt-Runden ≥ 12)
- [ ] JWT-Tokens kurzlebig (≤ 15 Minuten)
- [ ] Refresh-Tokens verwenden HTTP-only sichere Cookies
- [ ] Rate Limiting auf allen Endpoints (strenger bei Auth)
- [ ] Sicherheitsheader konfiguriert (CSP, CORS, HSTS)
- [ ] SQL-Abfragen parametrisiert
- [ ] Abhängigkeiten aktuell (`npm audit`)
- [ ] Umgebungsvariablen nicht in der Versionskontrolle
- [ ] HTTPS überall erzwungen
- [ ] Fehlermeldungen geben keine internen Details preis
- [ ] Protokollierung erfasst Sicherheitsereignisse
- [ ] CORS korrekt konfiguriert (nicht `*` in Produktion)

### Fazit

Sicherheit ist kein Feature — es ist ein **Fundament**. Jede Codezeile, die du schreibst, sollte mit Sicherheit im Hinterkopf geschrieben werden. Die obigen Praktiken sind nicht erschöpfend, aber ihre Implementierung schützt dich vor den häufigsten Angriffsvektoren.

Denke daran: **Es ist immer günstiger, einen Sicherheitsvorfall zu verhindern, als sich davon zu erholen.**
