## なぜセキュリティが重要なのか

**情報セキュリティのポストグラデュエート・ディプロマ（大学院修了証）** を持つ開発者として、私は小さな脆弱性がどれほど壊滅的なデータ漏洩につながるかを目の当たりにしてきました。この記事では、すべての開発者が実施すべき基本的なセキュリティプラクティスについて解説します。

### OWASP Top 10 (2025年版)

OWASP Top 10は、Webアプリケーションにおけるセキュリティリスクの世界的な指標です：

1. **アクセス制御の不備 (Broken Access Control)** — 最も深刻な脆弱性
2. **暗号化の失敗 (Cryptographic Failures)** — 適切な暗号の実装不備
3. **インジェクション (Injection)** — SQL, NoSQL, LDAP, OSコマンドインジェクション
4. **安全でない設計 (Insecure Design)** — アーキテクチャ設計時の欠陥
5. **セキュリティ設定の不備 (Security Misconfiguration)** — 初期設定のままのアカウント、過剰なアクセス権限
6. **脆弱なコンポーネントの使用 (Vulnerable Components)** — アップデートされていない古いライブラリ
7. **識別と認証の失敗 (Authentication Failures)** — 脆弱なパスワード、セッションハイジャック
8. **ソフトウェアとデータの整合性の失敗 (Data Integrity Failures)** — CI/CDパイプライン改ざん
9. **ログとモニタリングの失敗 (Logging Failures)** — 不十分な監視体制
10. **サーバ側要求偽造 (SSRF)** — 外部APIへの不正なリクエスト

### 入力値の検証 — 最初の防御線

**ユーザーの入力は決して信用してはいけません。** クライアントとサーバーの双方で厳格に検証を行ってください：

```typescript
import { z } from "zod";

// 入力値に対する厳密なスキーマを定義する
const CreateUserSchema = z.object({
  email: z.string().email().max(255),
  password: z
    .string()
    .min(12, "パスワードは12文字以上である必要があります")
    .regex(/[A-Z]/, "大文字を1文字以上含める必要があります")
    .regex(/[a-z]/, "小文字を1文字以上含める必要があります")
    .regex(/[0-9]/, "数字を1文字以上含める必要があります")
    .regex(/[^A-Za-z0-9]/, "記号を1文字以上含める必要があります"),
  name: z
    .string()
    .min(2)
    .max(100)
    .regex(/^[a-zA-ZÀ-ÿ\s'-]+$/),
});

// コントローラーまたはハンドラー内での検証
async function createUser(req: Request) {
  const result = CreateUserSchema.safeParse(await req.json());

  if (!result.success) {
    return new Response(
      JSON.stringify({
        error: "バリデーションに失敗しました",
        details: result.error.flatten(),
      }),
      { status: 400 },
    );
  }

  // 安全なデータとして result.data を使用
  const { email, password, name } = result.data;
  // ... パスワードをハッシュ化してユーザーを登録する
}
```

### 認証のベストプラクティス

#### パスワードハッシュ化には bcrypt または Argon2 を使用する

```typescript
import bcrypt from "bcrypt";

const SALT_ROUNDS = 12;

async function hashPassword(password: string): Promise<string> {
  return bcrypt.hash(password, SALT_ROUNDS);
}

async function verifyPassword(
  password: string,
  hash: string,
): Promise<boolean> {
  return bcrypt.compare(password, hash);
}
```

> **絶対に** パスワードをプレーンテキストで保存しないでください。MD5やSHAなどの高速なハッシュアルゴリズムをパスワードに使用してはいけません。パスワード専用に設計された、検証時間調整可能なアルゴリズムを使用してください。

#### JWTのベストプラクティス

```typescript
import jwt from "jsonwebtoken";

// ✅ 推奨: 短命なアクセストークン + リフレッシュトークンのローテーション
const accessToken = jwt.sign(
  { userId: user.id, role: user.role },
  process.env.JWT_SECRET!,
  {
    expiresIn: "15m", // 短期間のみ有効
    issuer: "cristianarando.app",
    audience: "api",
  },
);

// ✅ 推奨: HTTP-only, Secure, SameSite属性を持たせたCookie
response.cookie("refreshToken", refreshToken, {
  httpOnly: true, // JavaScriptからアクセス不可（XSS対策）
  secure: true, // HTTPS通信でのみ送信
  sameSite: "strict", // CSRF保護
  maxAge: 7 * 24 * 60 * 60 * 1000, // 7日間有効
  path: "/api/auth", // 送信先エンドポイントの限定
});
```

### APIセキュリティチェックリスト

APIを構築する際は、必ず以下を導入してください：

#### レートリミット (接続頻度制限)

```typescript
import rateLimit from "express-rate-limit";

// API全体のレートリミット設定
const apiLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15分間
  max: 100, // 1回あたりの最大リクエスト数
  standardHeaders: true,
  legacyHeaders: false,
  message: {
    error: "リクエストが多すぎます。しばらく時間をおいてお試しください。",
  },
});

// 認証エンドポイントには厳格なリミットを設定
const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 5, // 15分間に5回まで試行可能
  skipSuccessfulRequests: true,
});
```

#### セキュリティヘッダー

```typescript
import helmet from "helmet";

app.use(
  helmet({
    contentSecurityPolicy: {
      directives: {
        defaultSrc: ["'self'"],
        scriptSrc: ["'self'", "'unsafe-inline'"],
        styleSrc: ["'self'", "'unsafe-inline'", "fonts.googleapis.com"],
        fontSrc: ["'self'", "fonts.gstatic.com"],
        imgSrc: ["'self'", "data:", "https:"],
        connectSrc: ["'self'"],
        frameSrc: ["'none'"],
        objectSrc: ["'none'"],
      },
    },
    crossOriginEmbedderPolicy: true,
    crossOriginOpenerPolicy: true,
    crossOriginResourcePolicy: { policy: "same-site" },
    referrerPolicy: { policy: "strict-origin-when-cross-origin" },
  }),
);
```

### SQLインジェクションの防止

**常にプレースホルダーを使用したパラメータ化クエリ（静的プレースホルダー）を使用してください:**

```typescript
// ❌ 危険 — SQLインジェクションに対して脆弱
const query = `SELECT * FROM users WHERE email = '${email}'`;

// ✅ 安全 — パラメータ化クエリを使用
const result = await db.query("SELECT * FROM users WHERE email = $1", [email]);

// ✅ 安全 — ORM (Prismaなど) の使用
const user = await prisma.user.findUnique({
  where: { email },
});
```

### デプロイ前のセキュリティチェックリスト

すべてのWebサイトを本番デプロイする前に、以下の項目を確認してください：

- [ ] すべての入力データが検証・サニタイズされているか
- [ ] パスワードはbcrypt/Argon2等で適切にハッシュ化されているか (ストレッチング回数が十分か)
- [ ] JWTの有効期限が短く設定されているか (15分以下を推奨)
- [ ] リフレッシュトークンはHTTP-onlyでSecureなCookieに保存されているか
- [ ] 各エンドポイントにレートリミットが設定されているか (特にログイン機能)
- [ ] セキュリティヘッダー(CSP, CORS, HSTS)が適切に設定されているか
- [ ] SQLクエリがすべてパラメータ化されているか
- [ ] 依存ライブラリが最新であるか (`npm audit`で確認)
- [ ] 環境変数がリポジトリに含まれていないか
- [ ] HTTPS接続が強制されているか
- [ ] エラーメッセージからシステムの内部情報が漏洩していないか
- [ ] ログ監視体制が整っており、監査ログを取得できているか
- [ ] 本番環境におけるCORSが適切に制限されているか (`*`になっていないか)

### おわりに

セキュリティは単なる「追加機能」ではなく、システムの **「土台」** です。あなたが書くすべてのコードは、最初からセキュリティを考慮して書かれるべきです。上記の対策ですべてを防げるわけではありませんが、一般的な攻撃の大部分を防ぐことができます。

覚えておいてください。 **「漏洩事故からの復旧コストに比べれば、事前の対策コストははるかに安い」** のです。
